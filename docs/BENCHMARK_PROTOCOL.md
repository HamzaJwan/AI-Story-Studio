# Benchmark Protocol

## القاعدة الأساسية

**لا يدخل أي AI media engine (TTS / Image / Video) في المنتج إلا إذا مرّ بـ Benchmark Gate ونجح فعلياً على عتادنا الحقيقي.**

لا قرار يُبنى على:
- README فقط.
- اسم مشهور أو شائع.
- نتائج منشورة على عتاد مختلف.
- افتراضات عن VRAM/driver/CUDA من الذاكرة.

كل قرار يُبنى على:
- تجربة فعلية على AI Server (أو الجهاز المستهدف فعلياً للتشغيل).
- ملف output حقيقي محفوظ.
- قياس زمني ومواردي حقيقي.
- توثيق في هذا الملف أو في ملف benchmark خاص بالـ engine (مثل `docs/SILMA_TTS_BENCHMARK.md`).

راجع `docs/HARDWARE_PROFILE.md` قبل أي benchmark لمعرفة حدود العتاد الفعلية.

## Benchmark Gate — الحقول المطلوبة

كل benchmark engine جديد (TTS أو Image أو Video) يجب أن يسجّل الحقول التالية، في جدول أو ملف منفصل واضح:

| الحقل | الوصف |
| --- | --- |
| `engine` | اسم الـ engine (مثل SILMA, AllTalk, ComfyUI, WanGP) |
| `version` | رقم إصدار / Docker image tag / git commit المستخدم فعلياً |
| `hardware` | السيرفر/الجهاز الذي تم التشغيل عليه (مرجع إلى `HARDWARE_PROFILE.md`) |
| `command` | الأمر الفعلي المستخدم للتشغيل (docker run / docker compose / script) |
| `input_sample` | وصف أو مسار لعينة الإدخال المستخدمة فعلياً |
| `output_sample_path` | مسار ملف الناتج الفعلي (WAV/MP3/PNG/MP4) — لا تقبل benchmark بدون ناتج محفوظ |
| `cold_start_time` | زمن أول تشغيل (يشمل تحميل الموديل/bootstrap) |
| `warm_run_time` | زمن التشغيل بعد التحميل الأول |
| `vram_usage` | استهلاك VRAM الفعلي من `nvidia-smi` أثناء التشغيل |
| `cpu_ram_usage` | استهلاك CPU/RAM إن توفر قياسه |
| `quality_notes` | ملاحظات جودة حقيقية (وضوح صوت، تشويه صورة، تماسك فيديو...) |
| `known_limitations` | قيود معروفة (حجم bootstrap، حاجة GPU معين، رخصة صوت...) |
| `verdict` | واحدة فقط من: `PASS` / `CANDIDATE` / `BLOCKED` / `REJECTED` |

### تعريف Verdict

- **PASS** — يعمل فعلياً على عتادنا، الناتج مقبول، يمكن اعتباره lab ناجح (لا يعني دمجاً فورياً في المنتج).
- **CANDIDATE** — يعمل جزئياً أو يحتاج شرطاً إضافياً (مثل خدمة منفصلة، رخصة صوت، تحسين زمن) قبل اعتباره PASS.
- **BLOCKED** — فشل تقني حالياً (تثبيت/تعارض dependencies/عتاد غير كافٍ) لكن قد يُعاد تجربته لاحقاً بعد تغيير.
- **REJECTED** — تم تجربته وقرار صريح بعدم استخدامه (جودة ضعيفة جداً أو زمن غير عملي أو مخاطر ترخيص).

أي engine بحالة غير `PASS` **لا يدخل المنتج**، بدون استثناء.

## دور البحث المجتمعي (Web Research)

قبل أي benchmark، يمكن استخدام البحث (official docs, GitHub issues/discussions, community examples, تجارب على RTX 4060/4060 Ti أو بطاقات 8GB/16GB VRAM, تجارب Arabic TTS أو image/video generation) لتكوين **shortlist** فقط.

- البحث يعطي مرشحين محتملين ويوفر وقتاً (مثل معرفة مشاكل معروفة قبل التجربة).
- البحث وحده **لا يكفي لاتخاذ قرار دمج**. القرار النهائي يُبنى فقط على نتيجة benchmark فعلية مسجّلة بالحقول أعلاه.
- لا تُغيّر خطة المنتج بناءً على نتائج بحث نظري فقط.

## أنواع الـ Benchmark

### TTS Benchmark

يجب أن يشمل:
- نص عربي قصير ونص عربي طويل.
- جملة تحتوي علامات ترقيم وأرقام.
- قياس سرعة التوليد (cold + warm).
- تقييم جودة النطق العربي.
- هل يحتاج reference voice؟ (راجع قاعدة الأصوات في `docs/TTS_ENGINE_BENCHMARK_MATRIX.md` — لا أصوات أشخاص حقيقيين أو مشاهير بدون إذن).
- هل يدعم Docker فعلياً على عتادنا؟
- هل يستخدم GPU فعلياً (تأكيد من `nvidia-smi` لا افتراض)؟
- هل الناتج WAV/MP3 صالح ويُفتح بدون أخطاء؟

### Images Benchmark (لاحقاً فقط — لا تبدأ قبل Phase 0.5 Gate)

- لا يدخل ComfyUI أو أي محرك صور في المنتج قبل lab ناجح.
- اختبار resolution واقعي (مثل 768x768 أو 1024x1024) — لا قياس على resolution مصغّر فقط.
- تسجيل الزمن وVRAM فعلياً.
- حفظ عينة output حقيقية.
- رفض الـ engine إذا كانت الجودة ضعيفة أو الزمن غير عملي على عتاد AI Server الفعلي.

### Video Benchmark (لاحقاً فقط — لا تبدأ قبل Phase 0.5 Gate)

- اختبار أولي بـ 3-5 ثوانٍ فقط، resolution منخفض/متوسط.
- تسجيل الزمن وVRAM فعلياً.
- رفض workflow يحتاج ساعات لمشهد واحد أو يعطي نتيجة غير مقبولة بصرياً.

## ملاحظة

هذا الملف **بروتوكول عملية فقط** — لا يحتوي نتائج. النتائج الفعلية تُسجَّل في ملفات benchmark مخصصة (مثل `docs/SILMA_TTS_BENCHMARK.md`) وتُلخَّص في `docs/TTS_ENGINE_BENCHMARK_MATRIX.md` أو ما يعادلها لاحقاً للصور والفيديو.
